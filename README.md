# GoGreen@APU

> A campus sustainability platform for **Asia Pacific University of Technology & Innovation (APU)** — built to make eco-friendly events discoverable, attendance trackable, and green participation rewarding.

GoGreen@APU connects students with environmental initiatives on campus through a gamified event system. Students earn **AP Coins** by attending events and engaging with the community, then redeem them for rewards. Every event is mapped to one or more of the **17 UN Sustainable Development Goals (SDGs)** so participants can see exactly which sustainability outcomes their activities contribute to.

## Overview

GoGreen@APU is a full-stack PHP/MySQL web application that serves as the central hub for sustainability-related events at APU. It supports the full event lifecycle — from event creation and approval, through registration and attendance, to certificate issuance and reward redemption.

The platform is designed around four roles (**Admin**, **Organizer**, **Collaborator**, **Student**) plus a **Guest** view for the public, each with its own dashboard and feature set.

### Core Concepts

| Concept | Purpose |
|---------|---------|
| **AP Coins** | In-app reward currency. Earned by attending events and posting comments. Spent on rewards from the catalog. |
| **SDG Mapping** | Each event tags itself with the relevant UN Sustainable Development Goals (1–17). Students can filter events by SDG. |
| **Attendance Codes** | 3-digit, time-limited (2-minute TTL) codes generated by Organizers/Admins for student check-in. |
| **Approval Workflow** | Events created by Organizers require Admin approval before they go live; student registrations require Organizer approval. |
| **Certificates** | PDF participation certificates generated per event, signed by an uploaded signature image. |

## User Roles

The system supports **4 authenticated roles** plus an unauthenticated **Guest** view.

### 👤 Guest (Unauthenticated)

The public-facing entry point. Anyone visiting the site can:

- Browse approved events (read-only)
- Read event details
- View About Us, FAQ, Contact Us, and Privacy Policy pages
- View certificates by lookup
- Register a new student account or log in

### 🎓 Student

The primary end-user — APU students who participate in sustainability events.

**Profile:** stores `course`, `graduation` year, and `ap_coins` balance.

**Capabilities:**
- Browse and filter events on the **Explore** page
- Register for events (with optional receipt upload for paid events)
- Cancel registration before the deadline
- Use attendance codes to check in at events
- Like events and post comments (+5 AP Coins per comment)
- View **My Events** — upcoming and past
- Download participation certificates as PDFs
- Browse the **Rewards** catalog and redeem rewards using AP Coins
- Receive in-app notifications (system, event, reward)
- Manage personal account settings (avatar, password, security questions)

### 🧑‍💼 Organizer

Student club leaders or staff who create and run events. Each Organizer is linked to a single Club.

**Profile:** stores `club_id` and `positions` within the club.

**Capabilities:**
- Create events with full metadata: title, description, dates, location, embedded map, transportation details, pricing, max participants, registration deadline, AP Coins reward, and SDG tags
- Edit or delete their events; replace the event poster
- View participants list and approve/deny pending registrations
- Mark attendance manually
- **Generate attendance codes** (3-digit, 2-minute TTL)
- Send announcements with optional images to all event participants
- Upload a signature image for that event's certificates
- View per-event reports

### 🤝 Collaborator

External partners or NGOs assisting with specific events. Scoped to the events they're attached to.

**Profile:** stores `event_id`, `type` (internal/external), `positions`, and optional `ngo_name`.

**Capabilities:**
- View overview of assigned event(s)
- View participant list
- View and generate attendance codes for assigned events
- Edit limited event details
- Manage their account / profile settings
- Access support pages

### 🛡️ Admin

System super-user with global access.

**Capabilities:**
- **User Management** — create, edit, delete Students / Organizers / Collaborators
- **Club Management** — CRUD operations on clubs (logo, description, email, website)
- **Event Management** — create events directly, delete any event, approve organizer-submitted events, mark participants, manage attendance codes
- **Rewards System** — create rewards, manage inventory & pricing, toggle availability, view and fulfill redemption requests
- **Certificates** — configure certificate signature per event
- **System Settings** — update site logo/icon, manage pinned posts, manage FAQs
- **Reports** — generate activity reports across all roles
- **Audit Trail** — every privileged action is recorded in the `logs` table

## Database Schema

Database name: **`rwdd`** — full SQL dump in [GoGreen@APU.sql](GoGreen@APU.sql).

### Core Tables

| Table | Description |
|---|---|
| `users` | All accounts (any role). Stores `apkey` (TP number), hashed password, role, profile fields, security questions/answers, created_at, last_login. |
| `students` | Student profile extension — `course`, `graduation`, `ap_coins`. |
| `organizers` | Organizer profile extension — `club_id`, `positions`. |
| `collaborators` | Collaborator profile extension — `event_id`, `type` (internal/external), `positions`, `ngo_name`. |
| `clubs` | Student clubs — name, description, logo, email, website. |
| `events` | Event listings — schedule, location, map embed, transport, pricing, capacity, registration deadline, AP coin reward, SDG flags 1–17, approval status, contact info. |
| `event_participants` | Registrations — links user ↔ event, with approval status, attendance status, receipt path, coins earned, check-in time. |
| `announcements` | Targeted messages per event with optional image. |
| `attendance_codes` | Time-limited 3-digit codes (2-minute TTL) for event check-in. |
| `certificates` | Per-event certificate signature. |
| `comments` | Event comments — each awards 5 AP Coins by default. |
| `notifications` | Per-user notifications, typed (system / general / event / reward). |
| `rewards` | Reward catalog — title, image, AP Coin cost, quantity, active flag. |
| `reward_redemptions` | Redemption history with status, fulfilled_by, fulfilled_at. |
| `faq` | FAQ entries grouped by category (general / events / account / rewards). |
| `logs` | Audit trail — user_id, role, action, details, date_time. |

### Key Relationships

```
users ─┬─ students (1:1)
       ├─ organizers (1:1) ── clubs (M:1)
       └─ collaborators (1:M) ── events (M:1)

clubs ── events (1:M)
events ─┬─ event_participants ── users
        ├─ announcements
        ├─ attendance_codes
        ├─ certificates
        └─ comments ── users

rewards ── reward_redemptions ── users
users ── notifications
users ── logs
```

```
/auth/login.php
   └─→ /actions/login_process.php
         └─→ redirect_role.php
               ├─ admin        → /frontend/admin/index.php
               ├─ organizer    → /frontend/organizer/my_events/index.php
               ├─ collaborator → /frontend/collaborator/index.php
               └─ student      → /frontend/student/index.php
```

### Student Registration Flow

```
/auth/register.php (4-step form: account → personal → security → review)
   └─→ /actions/register_process.php
         └─→ inserts into `users` + `students`
               └─→ redirect to login
```

### Event Lifecycle

```
Organizer creates event
   → status = pending
   → Admin reviews / approves
   → event becomes visible on Explore
   → Student registers
   → Organizer approves registration
   → On event day: Organizer generates attendance code (2-min TTL)
   → Student enters code → check-in recorded → AP Coins awarded
   → Certificate becomes available for download
```

## Installation & Setup

### Prerequisites

- **PHP 8.0+** (8.0.26 confirmed)
- **MySQL 5.7+** (5.7.44 confirmed)
- **Apache** (XAMPP, MAMP, or any LAMP/WAMP stack)
- A modern web browser

### Local Setup with XAMPP

1. **Clone the repository** into the XAMPP `htdocs` directory:

   ```bash
   cd /Applications/XAMPP/xamppfiles/htdocs       # macOS
   # or  C:\xampp\htdocs                          # Windows
   git clone https://github.com/AaronKhoolb/GoGreen-APU.git
   ```

2. **Start Apache and MySQL** from the XAMPP control panel.

3. **Create the database**:

   ```sql
   CREATE DATABASE rwdd CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
   ```

4. **Import the schema**:

   ```bash
   mysql -u root -p rwdd < GoGreen@APU.sql
   ```

   Or import `GoGreen@APU.sql` via phpMyAdmin.

5. **Configure database credentials** — edit [includes/db_conn.php](includes/db_conn.php):

   ```php
   $servername = "localhost";
   $username   = "root";
   $password   = "";          // your MySQL password
   $dbname     = "rwdd";
   ```

6. **Set upload directory permissions** (Linux/macOS):

   ```bash
   chmod -R 775 uploads/ assets/images/profile/
   ```

7. **Visit the app** at: `http://localhost/GoGreen-APU/`

## Authentication & Security

### Login

Authentication is performed against the `users` table on the `apkey` field (the student's APU TP number, e.g. `TP083872`).

```
auth/login.php
   ↓ POST
actions/login_process.php
   ↓ password_verify() against hashed password
   ↓ session populated: $_SESSION['user_id'], $_SESSION['role'], $_SESSION['apkey']
   ↓ logs the login event into `logs`
includes/redirect_role.php
   ↓ routes to the role's home dashboard
```

### Password Storage

Passwords are hashed with PHP's [`password_hash()`](https://www.php.net/manual/en/function.password-hash.php) using `PASSWORD_DEFAULT` (currently bcrypt) and verified with `password_verify()`. Plaintext passwords are never stored.

### Password Reset

Self-service via security questions. During registration the student picks **3 security questions** and supplies answers. The reset flow at [auth/reset-pass.php](auth/reset-pass.php) requires:

1. The TP number
2. Correct answers to all 3 security questions
3. A new password (and confirmation)

### Session Management

Configured in [includes/session.php](includes/session.php):

| Setting | Value |
|---|---|
| Cookie lifetime | 86400 seconds (24h) |
| `HttpOnly` | `true` |
| `SameSite` | `Strict` |
| `Secure` | `false` (set to `true` behind HTTPS in production) |

### Access Control

Every protected page begins with `includes/auth_check.php`, which validates:

1. A session exists (`$_SESSION['user_id']`)
2. The session role matches the path prefix (`/frontend/admin/` requires `admin`, `/frontend/organizer/` requires `organizer`, etc.)

Failed checks redirect to the login page or to the user's correct dashboard.

### Audit Logging

Every privileged action is recorded in the `logs` table (`user_id`, `role`, `action`, `details`, `date_time`). This includes login/logout, event create/update/delete, user management, reward redemptions, and approvals.

## Gamification: The AP Coins System

AP Coins (`students.ap_coins`) are the in-app reward currency.

### Earning Rules

| Action | AP Coins Awarded |
|---|---|
| Attending an approved event | Configured per event (`events.coins_earned`) |
| Posting a comment on an event | **+5** (default in `comments.coins_awarded`) |
| Liking an event | 0 (engagement metric only — boosts `events.like_count`) |

Coins are awarded **on attendance check-in** (when a valid attendance code is submitted), not at registration. This prevents farming by mass-registering and never showing up.

### Redemption Flow

```
Student → Rewards page → "Redeem"
   ↓ checks: reward.is_active = 1 AND reward.quantity > 0 AND student.ap_coins >= reward.cost
   ↓ creates a row in reward_redemptions (status = 'pending')
   ↓ deducts cost from student.ap_coins
   ↓ Admin marks as fulfilled in /frontend/admin/rewards/reward_redemption.php
```

### Anti-Abuse Notes

- Comments are gated by registration (only registered participants of an event can comment).
- Attendance codes expire 2 minutes after generation, so a leaked code is useless once stale.
- Manual attendance marking by Admin/Organizer is logged.

## Attendance Code Mechanics

A defining feature of the platform — short-lived codes prevent fake check-ins.

```
Organizer / Collaborator / Admin
   ↓ POST /actions/admin/event/get_attendance_code.php
       (or organizer's equivalent)
   ↓ generates random 3-digit code
   ↓ inserts into attendance_codes:
       (event_id, code, generated_by, generated_at = NOW(), expires_at = NOW() + 2 minutes)
   ↓ returns code as JSON

Student
   ↓ enters code on /frontend/student/attendance/index.php
   ↓ server validates: code matches AND expires_at > NOW() AND student is a registered participant
   ↓ on success: event_participants.status = 'attended', check_in_time = NOW(), AP Coins awarded
```

> A **new code** is generated each time the organizer presses the button — the previous code is invalidated.

## Notifications System

The `notifications` table delivers in-app messages to users. Notifications are typed:

| Type | Triggered when… | Example |
|---|---|---|
| `system` | Admin posts a global announcement | "Maintenance scheduled for Sunday 2am." |
| `general` | Account-level events | "Your password was changed." |
| `event` | Event-related changes | "Your registration for *Beach Cleanup* is approved." |
| `reward` | Reward redemption updates | "Your reward redemption is fulfilled — pickup at office." |

Each notification has an optional `link_event_id` so clicking it deep-links to the relevant event detail page. The bell icon in the navbar polls `actions/student/notification/get_notifications.php` and shows a badge for unread entries.

## Troubleshooting

### `Database connection failed`

The credentials in [includes/db_conn.php](includes/db_conn.php) are placeholders. Update them to point at your local MySQL instance.

### `404 / page not redirecting`

The app expects to be served from `http://localhost/GoGreen-APU/` (i.e. the project folder name is `GoGreen-APU`). If you renamed the folder, update absolute paths in [index.php](index.php) and any `window.location.href` references.

### Uploaded files don't appear

Check filesystem permissions on `uploads/` and `assets/images/profile/`. On Linux/macOS:

```bash
chmod -R 775 uploads/ assets/images/profile/
```

On Windows under XAMPP this is rarely an issue.

### Sessions don't persist

If you're testing on a non-`localhost` domain, the cookie `Secure` flag may need to be enabled. Set it in [includes/session.php](includes/session.php) and access the site over HTTPS.

### Attendance code "expired" immediately

Confirm the database server's timezone matches the PHP server's timezone. The code expiry uses `NOW() + INTERVAL 2 MINUTE` on MySQL, so a clock skew between PHP and MySQL will misvalidate.

### `head.php` missing

`includes/head.php` is in `.gitignore`. If you cloned the repo and the file isn't there, recreate it from the template — it bootstraps sessions, includes `db_conn.php`, defines the `<head>`, and pulls in global CSS/JS.

## Contributing

Contributions are welcome once a license is in place. Suggested workflow:

3. Follow existing conventions:
   - PHP files start with an author/date/description comment block.

## Frequently Asked Questions

**Why PHP and vanilla JS in 2026?**
The project began as university coursework with constraints on the stack. Familiar tooling kept the focus on the sustainability domain rather than the framework. A future rewrite (Laravel / Next.js) is a reasonable contribution.

**Where do AP Coins "come from"?**
They're synthetic — the database mints them when a student attends an event or comments. There's no real-money backing.

**Can the platform be used outside APU?**
Yes — the only APU-specific assumption is the `apkey` field formatted as a TP number. Replace that with email or a generic ID and most of the system carries over to any campus.

**Is this production-ready?**
Treat it as an educational reference and a working MVP. Before deploying to a real audience, address the items in the [Pre-Open-Source Checklist](#pre-open-source-checklist) and the [Roadmap](#roadmap--ideas-for-contributors).
